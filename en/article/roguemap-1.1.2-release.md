# RogueMap 1.1.2 released: RogueMemory adds namespace guard operation, making multi-tenancy more secure

> In a multi-user scenario, could a `delete(id)` delete the wrong memory of others? 1.1.2 Use namespace guards to prevent unauthorized operations from the API level.

---

## Update overview

1.1.2 is an enhanced version of RogueMemory’s API. The core change is to add namespace-level operating capabilities for multi-tenant scenarios. If you are working on an AI Agent multi-user system, this update directly solves the **data security** issue.

| New API | Description |
|----------|------|
| `exists(id)` | Efficiently determine whether memory exists without reading the complete record |
| `exists(id, namespace)` | Determine whether the memory under the specified namespace exists |
| `delete(id, namespace)` | Deletion with namespace guard, silently skipping if namespace does not match |
| `update(id, namespace, newContent)` | Update with namespace guard, silently skipping if namespace does not match |
| `deleteByNamespace(namespace)` | Batch delete all memories by namespace |

In addition, the backward compatibility issue of `OrdinalRegistry` deserialization has been fixed, and the data files generated by 1.1.1 can be seamlessly read after the upgrade.

---

## Problem background

In a multi-user AI Agent system, each memory is isolated by user through a namespace. The previous version provided basic namespace retrieval capabilities, but the write-side operations (delete, update) lacked namespace verification:

```java
// 1.1.1 — Can only operate by id, no namespace verification
mem.delete(id);          // If the ID is guessed or transmitted incorrectly, other users' data may be deleted.
mem.update(id, content); // Same as above
```

This requires additional maintenance of permission verification logic at the business layer. Once omitted, there is a risk of unauthorized access.

---

## Namespace guard

1.1.2 Added namespace guard parameters for delete and update operations to ensure that operations will not cross namespaces from the API level:

```java
// Only the memory under the user-A namespace will be deleted. If the id does not belong to the namespace, it will be silently skipped.
mem.delete(id, "user-A");

// Only update memory under user-A namespace
mem.update(id, "user-A", "Updated content");
```

The internal implementation is to first read the record verification namespace, then perform the operation if it matches, and return directly if it does not match. The business layer does not need to make additional permission judgments.

**Typical usage**: The business layer gets the current user ID (such as `userId`) from the session context, and all write operations carry this ID:

```java
String userId = getCurrentUserId();

// Specify namespace when saving
mem.add("User preference dark mode", Map.of(), userId);

// All subsequent operations are performed with namespace guards.
mem.update(memoryId, userId, "Users strongly prefer dark mode");
mem.delete(memoryId, userId);

// One-click cleanup when user logs out
mem.deleteByNamespace(userId);
```

---

## Batch deletion: deleteByNamespace

Added `deleteByNamespace(namespace)` method to delete all memories under the specified namespace. Typical scenario:

- **User Logout / GDPR Compliance** — Users request deletion of all personal data, all done with one command
- **Business Line Cleanup** — A certain business line goes offline and all its memories are cleared.
- **Test Environment Reset** — Clear data in the test namespace

```java
mem.deleteByNamespace("user-123");
```

All entries are traversed internally and soft deleted one by one, and the space is reclaimed during compaction.

---

## Existence check: exists

New `exists(id)` and `exists(id, namespace)` methods are more efficient than `get(id) != null` - no need to read the complete record, only check the index:

```java
// Simple existence check
if (mem.exists(memoryId)) {
    // memory exists
}

// Existence check with namespace
if (mem.exists(memoryId, userId)) {
    // The memory exists and belongs to the current user
}
```

## Upgrade method

Just change the Maven dependency version number to `1.1.2`. The data files are backward compatible and do not need to be migrated:

```xml
<dependency>
    <groupId>com.yomahub</groupId>
    <artifactId>roguemap-core</artifactId>
    <version>1.1.2</version>
</dependency>

<dependency>
    <groupId>com.yomahub</groupId>
    <artifactId>roguemap-memory</artifactId>
    <version>1.1.2</version>
</dependency>
```
